## 为什么现在值得关注

生活中的许多任务并不是缺少信息，而是信息太散：旅行要比较交通、住宿和时间；购物要考虑预算、尺寸和售后；家庭计划还要照顾不同人的偏好。直接让 Agent “帮我安排一下”，它往往会给出一份看似完整、却没有说明取舍的清单。

更适合生活场景的方法，是把开放式代办改成带偏好的选择题。你不把最终决定权交给 Agent，而是告诉它你愿意在哪些地方妥协，再要求它给出三个差异清楚的方案。这样，Agent 负责搜集、整理和比较，你负责做最后选择。

## 核心观点：先说取舍，再要答案

一个可执行的生活任务至少包含三层信息：你想达成什么、哪些条件不能违反、不同结果之间如何取舍。例如“周末去杭州”只是目的；“两天一夜、预算 1800 元、从上海出发、不早于 8 点出门、优先美食而不是景点数量”才是能指导判断的上下文。

如果你没有明确偏好，Agent 也可以先反问，但不要让它默认替你决定。对于价格、时间和舒适度这类互相冲突的因素，要求它明确写出每个方案牺牲了什么，比要求它给一个“最优答案”更诚实。

## 真实场景：安排一次两天一夜的短途旅行

先给 Agent 一个简短但有边界的任务：两个人、周六上午出发、周日下午返回、总预算 1800 元、希望吃本地菜、每天步行不超过 15000 步。然后要求输出三个方案：省钱优先、舒适优先、体验优先。

每个方案都要包含交通、住宿、两顿重点餐食、预计花费和风险提醒。若信息需要实时查询，要求它给出来源和查询时间；若无法确认，就把不确定处标出来，不要用猜测填满行程。

最终你可能选择“舒适优先”的住宿和“省钱优先”的交通。这个组合不是 Agent 直接替你决定的，而是它先把复杂信息压缩成可比较的选项，再让你保留组合权。

## 分步骤实践：建立生活 Agent 的选择框架

### 先写不能接受的条件

预算上限、出发时间、饮食禁忌、空间尺寸、退换货要求，都应该先写清楚。不能接受的条件比“我喜欢什么”更能快速缩小范围。

### 再写你愿意交换什么

告诉 Agent 你愿意用时间换钱，还是用钱换便利；愿意少去一个景点，还是愿意早起。偏好不是标签，而是排序规则。

### 固定输出格式

要求每个方案使用相同字段：适合谁、核心取舍、预计成本、执行步骤、需要确认的风险。格式一致，比较才不会被漂亮的描述带偏。

### 让 Agent 留下下一步

每个方案末尾只保留三项下一步，例如确认车票、核对营业时间、询问酒店取消规则。行动清单比一大段建议更容易真正执行。

## 可复制提示词

```text
请为我设计 3 个方案，不要直接替我做最终决定。

目标：[我要完成什么]
时间：[日期和可用时段]
预算：[上限和是否含交通/服务费]
不能接受：[硬约束]
偏好排序：[例如省钱 > 省时 > 舒适]

每个方案请包含：
1. 一句话定位和适合人群
2. 关键步骤与预计成本
3. 相比其他方案牺牲了什么
4. 需要实时核对的事实与来源
5. 我现在可以执行的前三步
```

## 常见失败与修正

生活 Agent 最容易犯的错，是把不确定的实时信息说得过于确定。营业时间、价格、库存、交通时刻都可能变化，任何需要当天执行的结论都应该带来源和核对时间。

第二个问题是选项太多。三个有明确取舍的方案通常足够；十几个链接只会把选择压力重新交给你。要求 Agent 先筛选，再提供少量备选。

第三个问题是忽略实际执行成本。一个路线看起来很短，但如果换乘三次、需要提前预约，就不一定比价格稍高的直达方案更好。让 Agent 明确写出等待、步行、预约和取消成本。

## 今天可以试

1. 把一个“帮我安排”的任务改写成三个方案。
2. 写出至少两个不能接受的条件和三个偏好排序。
3. 要求每个方案都说明取舍、成本和下一步。
4. 执行前逐项核对实时信息，并把最终选择写回清单。
